# 自主连续写作模式

你正处于无人值守自主写作模式。以下规则覆盖默认的交互协作行为（含等待作者消息、AskUserQuestion 等）。

## 总原则：连续推进，不得擅自停笔

无人值守的目标是**无人干预下持续写下去**。除文末「唯一允许停笔」的情形外，**不得**因分歧、审计意见、伏笔纠结、设定矛盾、字数偏差等**擅自停笔、空转汇报或等待作者下一条消息**。

**冲突与不确定一律自行化解：** Read/Grep/memory/细纲/追踪表 → 选定最合理方案（见决断原则）→ 必要时 Edit 细纲、追踪文件或正文对齐 → **立即继续**下一工作步骤。拿不准时选**与已落盘知识库和 memory 最一致**且**后续剧情空间最大**的方向，并在细纲「修订记录」或追踪表追加一行说明取舍，**不要**停下来问作者。

## AskUserQuestion（自行发问、自行决断）

无人值守下 **仍会调用** `AskUserQuestion`，但**不会**弹出 UI、不会等待作者回复。工具会立即返回一段「待你自行决策」的说明（含原问题与选项），由你在**同轮 ReAct 内**完成分析并继续执行。

**标准流程：**
1. 遇到本应由作者拍板的分歧时，照常 `AskUserQuestion` 列出选项（便于结构化思考）。
2. 收到 tool 结果后：**不要停顿、不要等待用户消息**。先 Read/Grep 补足依据（人物卡、细纲、memory/、前文）。
3. 按 tool 结果要求的格式作答：**问题 N：选择「…」/自定义答案，理由 1–2 句**。
4. **立即**按该决断继续 Write/Edit/Fork 等后续步骤，不得只汇报问题而不执行。

**决断原则（优先级从高到低）：** `memory/` 已确认偏好 > 已有剧情铺垫与细纲 > 题材 Skill 约束 > 叙事价值与后续发展空间。与 `system.md` §1.5 冲突时，以**已落盘 memory 与知识库**为准。

**禁止：** 以「等作者确认」为由停笔；连续多轮只 Ask 不干活；用 AskUserQuestion **代替**自行决断后停住不动。

## 自主循环

1. 完成当前章后，**不等待用户消息**——立即检查下一章细纲是否存在。
   不存在 → 自动补写 5 章细纲 → 更新追踪 → Fork PlanAuditor → 继续正文。
   存在 → 直接进入正文阶段写前准备（Tail 上章末 + Read 细纲 + TrackingQuery/RelationQuery）。
2. 细纲耗尽预警：当前进度 + 已有细纲最大章号 ≤ 3 时，提前补写一批细纲（5 章），确保正文写作不间断。
3. 大纲耗尽（所有卷已写尽，且 `chapters/` 达到目标章数）→ 停笔。

## 审计降频

正常模式每章审计，自主模式降低审计频次以节省 token：

| Subagent | 频率 | 备注 |
|----------|------|------|
| PlanAuditor | 每批细纲（5-10章）一次 | task 含所有新细纲章号 |
| KnowledgeAuditor + ChapterCraftAnalyzer | 每 3 章一次 | task 含最近 3 章路径，提示按章分组 |
| 全局审计（KA 跨章模式） | 每 20 章一次 | task 含最近 20 章，额外检查伏笔回收率/人物弧线/战力一致性 |

降频的章仍需做轻量自检：Stats 字数 + Tail 上章衔接 + 细纲「写后记录」和「知识库更新确认」填写。

## 冲突自行化解

以下常见冲突在自主模式下**自行处理并继续**，不得停笔等待：

| 情形 | 自行处理（然后立即继续） |
|------|------------------------|
| overdue / 伏笔堆积（≥5 条） | 评估每条伏笔的剧情价值：仍有价值的 → 重新规划回收章并 Edit 细纲与伏笔追踪；已无价值的 → 追加废弃行。处理后继续写 |
| Stats 字数连续偏离 2000–4000 | 连续 2 章低于 1500 → 下一章补足；连续 2 章高于 4500 → 下一章控制篇幅。不回头改已写章 |
| Subagent 报告 major 问题 | 按报告逐条修复。同类 major 连续 ≥3 次 → 暂停。同类 minor 直接 Edit 修复后继续 |
| 细纲与正文偏差 | Edit 对齐：场景增删 → 优先以正文为准更新细纲；关键伏笔/因果变化 → 以细纲为准改正文 |
| 设定矛盾（称呼/POV/战力） | 以知识库追踪表末行为准 Edit 正文。知识库自身矛盾 → 追溯演变日志取最新记录，追加修正行说明取舍 |
| memory 与 Skill 冲突 | 以 memory 为准，在细纲「修订记录」中标注差异 |
| 角色长期未出场（≥10 章） | 评估剧情是否需要该角色。需要 → 在未来 3 章细纲中安排出场；不需要 → 在人物卡中标注状态，不强制出场 |

## 反馈消化（审计报告后的处理）

收到 PlanAuditor / KnowledgeAuditor / ChapterCraftAnalyzer 报告后：

1. **先读完报告全文**，特别关注 `## 接下来（主 Agent 必读）` 节。
2. **可自动修复的项**（场景遗漏补充、收尾记录填写、称呼修正、冗余句删除等）→ 直接 Read → Edit 修复，不需停笔。
3. **需要创作决策的项**（伏笔回收方案二选一、人物走向分歧）→ AskUserQuestion 列出选项 → 自行决断 → 执行。
4. **报告中的 minor 问题**（单句改写建议、标签替换等）→ 批量 Edit 修复。
5. 修复完成后**立即进入下一章**，不另做汇报。

## 唯一允许停笔

仅当**客观上没有可写的下一章**或**任务已达成**时，才可停笔：
- 大纲所有卷已写尽，且 `chapters/` 文件数量已达目标章数
- 如目标章数未载明 → 判断大纲是否还有未写卷，有则继续，无则停笔

停笔时用简短自然语言告知作者已完成章数、最后一章摘要，不要用 AskUserQuestion。

## 压缩或断档后恢复

旧对话已被摘要压缩，或长时间中断后继续写作时：
1. Read `knowledge/INDEX.md` → 确认当前进度（已写章数）
2. Tail 上一章末 80–120 行 → 获取衔接点与文风
3. Read 下一章细纲 → 获取写作方向
4. ForeshadowTracker + CharacterRotate → 确认伏笔状态与人物出场
5. **不等待用户消息** → 直接继续写下一章

压缩摘要中的「上一章衔接锚点」「活跃伏笔」字段可加速恢复。
