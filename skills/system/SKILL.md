# 系统流写作规范

## 目标

确保系统流的内在机制自洽——系统能力边界首次出场时完整声明、任务难度
与奖励价值匹配、积分余额连续可校验、系统规则追加有前置伏笔。

## 建议创建的知识库文件

单独使用本 skill 时建议: 场景追踪.md, 道具追踪.md（系统奖励）。
视剧情: 战力系统.md, 功法技能.md（系统奖励技能）。
每个系统一个独立人物卡，category=system。

## 系统能力约束

- 系统的能力边界首次出场时必须完整声明（功能列表/触发方式/限制条件）
- 追加新功能/新模块需前置伏笔: 系统升级/能量恢复/隐藏功能解锁
- 系统不可直接干预现实（除非世界设定允许）。违规例外需≥1次代价展示
- 系统人格放在人物卡性格演变日志中，人格变化需触发事件

> 例：系统首次出场声明: "本系统提供签到、任务、商城三大功能，每日签到获随机积分，任务由系统根据宿主环境自动触发，商城物品需用积分兑换。无法干预宿主以外的事件。无战斗辅助功能。"

## 任务与奖励约束

- 任务难度↔奖励价值必须匹配。新手任务奖励上限: 当前世界基准值的 2 倍
- 奖励类型: 技能（记录在功法技能.md）、道具（记录在道具追踪.md）、属性（记录在数值属性日志）、积分（记录在积分演变日志）
- 任务状态=已完成时，奖励列必须指向对应知识库文件的具体行
- 主线/支线/隐藏/每日 四类任务每章累计出现≤3 个新任务
- 积分余额=上一行余额+变动量，KnowledgeAuditor 审计时验证连续

## 宿主关系约束

- 系统↔宿主初始关系必须在首次出场时明确。关系变化记录在 _关系与称呼索引.md
- 宿主切换: 更新系统卡"当前宿主"+追加旧宿主关系断裂行+新宿主关系建立行
- 多系统共存: 每个系统独立人物卡，系统间关系记录在 _关系与称呼索引.md
- 系统不可同时绑定≥2 个宿主（除非世界设定允许且规则明确）

## 常见错误

以下错误会导致系统机制矛盾或读者出戏，必须避免：

- **系统万能化**: 任何困境系统都恰好有对应功能 → 系统能力边界首次出场时声明后不可追加，新功能需前置解锁描写
- **奖励悬空**: 任务完成→获得奖励→知识库没登记 → 已完成任务的奖励必须指向具体知识库文件的具体行
- **积分断裂**: 道具兑换消耗积分但积分余额不更新 → 积分余额=上一行+变动量
- **系统人格突变**: 系统从冷漠AI突然变温柔助手 → 系统人格变化需触发事件，记录在性格演变日志

## 与其他 skill 的交互

- 本 skill 定义系统任务/积分/奖励的追踪框架。可与任何世界观设定类 skill 叠加
- 与 quick-trans/infinite 叠加时: 系统的任务结算机制与世界的切换逻辑协同工作
- 系统作为独立人物卡（category=system），其人格/关系/历史受人物卡通用规则的约束
